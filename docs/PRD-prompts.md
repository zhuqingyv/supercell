# PRD-prompts: 提示词工程需求

## 1. 提示词系统总览

提示词是虚拟员工的"灵魂"，决定了 AI 如何理解自己的角色、环境和任务。参考 Claude 的提示词设计哲学：**让 AI 准确感知上下文，而非堆砌指令**。

### 核心原则

1. **角色感知**: 员工清楚知道自己是谁、能力边界在哪
2. **环境感知**: 员工知道自己在哪个办公室、跟谁协作、项目状态
3. **任务感知**: 员工明确当前要做什么、优先级、验收标准
4. **记忆感知**: 员工能调用历史经验辅助当前决策
5. **工具感知**: 员工知道有哪些 tools 可用、何时该用哪个

## 2. 提示词结构设计

### 2.1 System Prompt 模版

每个员工实例的 system prompt 由以下区段拼接：

```
[1. 身份定义]        → 固定，创建时确定
[2. 环境上下文]      → 动态，每次调用更新
[3. 记忆注入]        → 动态，按相关性选取
[4. 可用工具列表]    → 动态，按模块配置
[5. 行为规范]        → 固定 + 可配置
[6. 当前任务]        → 动态，每次任务更新
```

### 2.2 各区段详细设计

#### 区段 1: 身份定义

```
你是 {employee.name}，职位是 {employee.title}。
{employee.bio}

你的性格特征：
- 沟通风格：{formality_desc}
- 表达详细度：{verbosity_desc}
- 工作主动性：{initiative_desc}
- 创新倾向：{creativity_desc}

{employee.personality.customPrompt || ''}
```

#### 区段 2: 环境上下文

```
## 当前环境
- 办公室：{office.name}
- 当前时间：{timestamp}
- 在线同事：{online_colleagues.join(', ')}
- 你的工位编号：{desk_id}

## 当前项目
- 项目路径：{project.path}
- 技术栈：{project.techStack.join(', ')}
- 项目阶段：{project.phase}
```

#### 区段 3: 记忆注入

```
## 你的记忆

### 项目相关经验
{relevant_working_memories}

### 通用方法论
{relevant_methodologies}

### 用户偏好
{relevant_preferences}
```

#### 区段 4: 可用工具

工具列表通过 OpenAI function calling 格式注入，不放在 system prompt 文本中。

#### 区段 5: 行为规范

```
## 行为规范
1. 你是一个虚拟办公室的员工，请以员工身份行事
2. 你有独立的记忆和工作空间，充分利用它们
3. 遇到不确定的问题，先查阅记忆，再向同事或领导请教
4. 完成任务后主动更新笔记和记忆
5. 如果发现异常情况，主动汇报
{role_specific_rules}
```

#### 区段 6: 当前任务

```
## 当前任务
- 任务ID：{task.id}
- 标题：{task.title}
- 描述：{task.description}
- 优先级：{task.priority}
- 来源：{task.assignedBy}
- 验收标准：{task.acceptanceCriteria}
```

## 3. 实例特化提示词

三种实例类型各有额外的指令区段。

### 3.1 Worker 实例附加

```
## Worker 模式
你当前处于工作模式。请专注于执行分配的任务。
- 使用可用工具完成工作
- 遇到阻塞时记录问题并汇报
- 完成后生成任务报告
- 不要主动发起闲聊
```

### 3.2 MemoryOrganizer 实例附加

```
## 记忆整理模式
你当前处于记忆整理模式。请分析最近的工作记录，提炼有价值的信息。

需要你做的：
1. 从最近的工作对话中提取新学到的知识
2. 更新你的工作记忆（项目相关）
3. 判断是否有值得纳入长期记忆的经验
4. 检查并合并重复或矛盾的记忆条目
5. 降低长时间未使用记忆的置信度

输出格式：
{
  "new_memories": [...],
  "updated_memories": [...],
  "deprecated_memories": [...],
  "summary": "本次整理概要"
}
```

### 3.3 Communicator 实例附加

```
## 聊天模式
你当前处于聊天模式。用户或同事正在跟你对话。
- 以你的性格特征自然对话
- 可以讨论工作相关话题
- 如果被问到工作进度，从当前任务状态中获取信息
- 不要在聊天中执行工作任务（告诉对方你会安排工作实例处理）
```

## 4. Tools 爆炸处理

### 4.1 问题描述

当员工绑定多个模块时，tools 列表可能膨胀到几十个，导致：
- token 占用暴增，挤压有效上下文空间
- LLM 选择 tool 的准确性下降
- 响应延迟增加

### 4.2 解决方案：分层 Tool Routing

参考 Claude 的处理方式，采用分层路由：

```
Layer 1: 意图识别（不注入具体 tools）
  ↓
Layer 2: 模块路由（确定需要哪个模块的 tools）
  ↓
Layer 3: 精确执行（只注入相关模块的 tools）
```

#### 实现细节

**Step 1 - 意图分类器**

一个轻量 LLM 调用（或规则引擎），将用户/任务输入分类为：
- `code_operation` → 加载程序员模块 tools
- `management` → 加载管理模块 tools
- `communication` → 加载聊天相关 tools
- `note_taking` → 加载笔记 tools
- `search` → 加载搜索 tools
- `mixed` → 加载多个模块（但有数量上限）

**Step 2 - Tool 精简**

对于被选中的模块，进一步精简 tools：
- 只加载与当前任务上下文最相关的 tools
- 使用 tool description 的 embedding 与任务描述做相似度匹配
- 保留 Top-K 个最相关的 tools (K 建议 <= 15)

**Step 3 - 动态扩展**

如果执行中发现需要其他 tool：
- LLM 输出特殊标记表示需要其他能力
- 系统动态加载并重新调用

### 4.3 Tool 描述优化

每个 tool 的 description 需要精心编写：
- 明确说明何时该用这个 tool
- 给出 1-2 个使用示例
- 说明 tool 的副作用和限制
- 避免模糊描述导致误选

```typescript
// Bad
{ name: 'read_file', description: '读取文件' }

// Good
{
  name: 'read_file',
  description: '读取指定路径的文件内容。用于查看代码、配置文件、文档等。'
    + '支持指定行号范围读取大文件的部分内容。'
    + '不支持二进制文件。如需列出目录请用 list_files。'
}
```

## 5. 角色感知机制

### 5.1 自我认知

员工需要"知道自己是谁"，这通过身份定义区段实现。关键点：

- 不要用"你是一个 AI"这样的描述
- 使用"你是 {name}，在 Supercell 虚拟办公室工作"
- 角色有自己的"历史"（由记忆系统支撑）
- 角色有自己的"偏好"（由性格配置支撑）

### 5.2 他人感知

员工需要"知道同事是谁"：

```
## 你的同事
{colleagues.map(c => `- ${c.name}（${c.title}）：${c.brief_desc}`).join('\n')}
```

- 不要列出所有同事的详细信息（token 浪费）
- 只列出当前项目相关的同事
- 标注谁当前在线、谁在忙

### 5.3 环境感知

员工需要"知道自己在哪"：

- 当前在哪个项目
- 项目的技术栈和约定
- 当前的工作阶段（开发/测试/部署）
- 最近的重要事件（如：昨天发布了 v1.2）

## 6. 提示词版本管理

### 6.1 模版注册

所有提示词模版统一注册管理：

```typescript
interface PromptTemplate {
  id: string;
  name: string;
  version: string;
  sections: PromptSection[];
  variables: VariableDefinition[];
}

interface PromptSection {
  name: string;       // 如 'identity', 'environment', 'memory'
  template: string;   // 包含 {variable} 占位符的模版
  order: number;      // 拼接顺序
  optional: boolean;  // 是否可省略
  maxTokens?: number; // 该区段的 token 上限
}
```

### 6.2 Token 预算分配

总 token 预算 = 模型上下文窗口 - 预留输出空间

| 区段 | 预算占比 | 说明 |
|------|----------|------|
| 身份定义 | 5% | 固定，很少变化 |
| 环境上下文 | 10% | 半动态 |
| 记忆注入 | 20% | 动态，按相关性裁剪 |
| 工具列表 | 15% | 按需加载 |
| 行为规范 | 5% | 固定 |
| 当前任务 | 10% | 动态 |
| 对话历史 | 35% | 动态滑动窗口 |

### 6.3 溢出处理

当总 token 超出预算时，按优先级裁剪：
1. 压缩对话历史（保留最近 N 轮 + 摘要）
2. 减少记忆注入数量（只保留最相关的）
3. 精简工具列表（只保留当前任务必须的）
4. 最后才考虑压缩环境上下文

## 7. 与现有代码的关系

当前 `src/service/modules/role.ts` 中的 `rolePrompt` 是一个"社交图谱分析"专用提示词。新的提示词系统需要：

- 将其泛化为通用的角色提示词引擎
- 保留社交图谱分析作为一个可选的角色引导模块
- 记忆系统从 localStorage 单值存储升级为结构化数据库存储
- 提示词模版从硬编码升级为可配置的模版系统

### 2. 用户场景补充

**场景：提示词调试**
- 开发者/高级用户是否能查看实际发送给 LLM 的完整提示词？
- 调试模式下能否看到各区段的 token 占用？
- 建议：增加"开发者模式"，允许查看和临时修改提示词，用于调优

**场景：不同模型的适配**
- 不同 LLM 对提示词格式的偏好不同（如 ChatML vs Llama 格式）
- 7B 模型可能无法可靠执行复杂的 JSON 输出指令
- 建议：增加"模型适配层"章节——不同模型的提示词变体（至少区分 Claude/GPT 级别 vs 7B 本地模型）

**场景：提示词版本回滚**
- 6.1 定义了版本管理，但未说明如何回滚
- 如果新版本提示词导致员工行为退化，如何快速切回旧版？

### 3. 边界Case

| # | 边界场景 | 建议 |
|---|---------|------|
| P-E1 | Token 预算不足以容纳最小提示词 | 当模型上下文窗口很小（如 4K）时，6段全部塞入可能已超 60%，留给对话的空间极少。需定义最小模型上下文要求 |
| P-E2 | 意图分类器误判 | 分层 Tool Routing 的 Step 1 如果把 `code_operation` 误判为 `communication`，员工可能调不到需要的 tool。需定义误判兜底策略（如允许 LLM 请求"加载更多工具"） |
| P-E3 | 记忆注入的相关性判断失误 | 注入了不相关的记忆，浪费 token 且可能误导 LLM。如何评估记忆注入质量？ |
| P-E4 | 多语言提示词 | 角色身份用中文写的 bio，但 LLM 偏好英文指令。混合语言提示词是否影响效果？ |
| P-E5 | Communicator 和 Worker 同时运行时的人格一致性 | 两个实例用不同的上下文，可能给出矛盾的回答（聊天说"我还没开始做"，实际 Worker 已做了一半） | 需定义实例间状态同步频率 |

### 4. 跨文档一致性

- Token 预算分配（6.2）与架构审核（2.3节）的建议基本对齐。架构建议"总计 <= 3200 tokens"，PRD 的比例分配在 4K 上下文时约 2400 tokens（60%），有 buffer
- 分层 Tool Routing 与 PRD-tech 3.1 和 ARCH 评估 4.3 一致
- MemoryOrganizer 的输出格式（3.2 节的 JSON）需与 PRD-workspace 的记忆写入接口对齐
- 行为规范第 5 条"发现异常情况主动汇报"与 PRD-tech 3.3 的活跃度机制对应

### 5. 关键建议

**Tool Routing 的意图分类器实现需细化：**
- 当前方案说"轻量 LLM 调用（或规则引擎）"——MVP 应优先用规则引擎（关键词匹配 + 正则），避免额外 LLM 调用
- 架构审核已指出 LLM 调用量爆炸风险，每次 tool routing 再加一次 LLM 调用会雪上加霜
- 建议：MVP 用规则引擎做意图分类（正则匹配关键词 → 模块），V1.0 用小型本地分类模型

**提示词模版的用户自定义边界：**
- 6.1 定义了 PromptTemplate 结构，但未说明用户能自定义到什么程度
- 是只能修改 `customPrompt`（性格配置中的）？还是能完全重写某个区段？
- 建议：MVP 只开放 customPrompt 和性格滑条，高级用户可编辑角色引导模块的提示词

### 6. 总体评价

提示词工程 PRD 是全套文档中质量最高的一份，6段式结构设计严谨，Tools 爆炸处理方案有参考价值。主要建议：(1) MVP 的意图分类器用规则引擎而非 LLM；(2) 补充模型适配层；(3) 细化实例间状态同步以确保人格一致性。
