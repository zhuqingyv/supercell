# PRD-overview: 产品总览

## 1. 产品定义

**产品名称**: Supercell (内部代号: gagent)

**一句话定义**: 一个以游戏化方式管理 AI Agent 团队的虚拟办公室桌面端应用。

**核心理念**: 给 Agent 提供一个虚拟的办公环境，用户以"养成游戏"的方式组建、培养、管理自己的 AI 团队，让 AI 员工像真人一样协作办公。

## 2. 产品目标

### 短期目标 (MVP)
- 用户能创建和管理多个 AI 员工角色
- 每个员工有独立的工作空间、记忆系统、聊天界面
- 程序员角色能执行 git/代码操作
- Q版像素风视觉呈现员工工作状态

### 中期目标 (V1.0)
- 完整的角色模版系统，一键创建标准员工
- 办公软件模块（虚拟 tools 注入）
- 员工之间能通力协作，自主分配任务
- 员工记忆系统成熟，能积累经验和方法论

### 长期目标
- 生态化：用户可分享和导入角色模版
- 多项目支持：不同项目可复用同一套团队
- Agent 真正"活跃"起来：自主发现问题、主动汇报

## 3. 目标用户画像

### 主要用户：独立开发者 / 小团队技术负责人
- 日常需要 AI 辅助编码、review、测试
- 希望 AI 能记住项目上下文和个人偏好
- 对 AI 协作有较高期望，不满足于单次对话
- 有一定技术背景，能部署本地 LLM

### 次要用户：AI 爱好者 / 效率工具玩家
- 对 AI Agent 概念感兴趣
- 喜欢养成类玩法
- 愿意尝试新工具来提升工作效率

### 用户核心诉求
| 诉求 | 描述 |
|------|------|
| 团队协作 | 不是一个 AI，而是一个团队在帮我干活 |
| 上下文记忆 | AI 记得我的项目、偏好、历史决策 |
| 可视化管理 | 直观看到谁在干什么，进度如何 |
| 低门槛 | 不需要写复杂配置，一键就能用 |

## 4. 项目范围

### 在范围内 (In Scope)
- 虚拟办公室主界面（像素风俯视图）
- 角色系统（创建/编辑/解雇员工）
- 模块系统（程序员模块、管理模块、办公软件模块）
- 工作空间（项目笔记、个人记忆、经验库）
- 聊天系统（员工间通信、用户与员工对话）
- 视觉系统（像素角色、骨骼动画、状态表现）
- 提示词引擎（角色感知、环境感知、tools 管理）
- 桌面端应用（React + bun + ts + 游戏引擎）

### 不在范围内 (Out of Scope - 当前阶段)
- 移动端适配
- 多用户协作（仅单用户本地使用）
- 云端部署和 SaaS 模式
- 自然语言以外的输入方式（语音、图片）

## 5. 与现有代码的关系

当前项目已有基础：
- **前端**: React 19 + Vite + Tailwind + Zustand 的聊天 UI（`src/`）
- **后端**: Express + SQLite 的 Agent 编排器（`server/`），已有 5 个 review Agent
- **服务层**: OpenAI SDK 封装，支持本地 LLM（LM Studio / Ollama）

需要演进的方向：
- 前端从纯聊天 UI 演进为游戏化虚拟办公室
- 后端 Agent 从"代码审查"演进为"虚拟员工"
- 角色系统从硬编码演进为用户可配置
- 新增游戏引擎层（Cocos / Phaser / PixiJS）渲染像素办公室

## 6. 成功指标

| 指标 | MVP 目标 |
|------|----------|
| 可创建角色数 | >= 5 个不同职位的员工 |
| 员工并发能力 | 单员工支持 3 实例并行（干活/整理记忆/聊天） |
| 任务完成率 | 程序员角色能独立完成简单 git 操作 |
| 记忆持久性 | 员工记忆跨会话保持，重启不丢失 |
| 渲染帧率 | 像素办公室场景 >= 30fps |

### 2. 用户场景缺失

PRD 缺少完整的用户场景描述，以下是需要补充的关键场景：

**场景1：新手引导**
- 用户首次打开应用时看到什么？空办公室？
- 是否有引导流程教用户创建第一个员工？
- 预设的模版和示例数据如何呈现？
- 建议：增加"新手引导"章节，定义首次使用流程（空办公室 -> 引导创建第一个员工 -> 分配第一个任务 -> 看到员工工作）

**场景2：日常使用**
- 用户每天打开应用的典型流程是什么？
- 员工在用户不在时是否自主工作？如何汇报进展？
- 用户如何快速了解"我离开后发生了什么"？
- 建议：增加"每日回顾"场景——进入办公室看到活动摘要、未读通知、任务进展

**场景3：高级用法**
- 批量操作（同时给多个员工分配相关联的任务）
- 自定义模块开发和导入
- 项目切换（多项目场景下如何管理）

### 3. 边界Case

| # | 边界场景 | 当前PRD是否覆盖 | 建议 |
|---|---------|----------------|------|
| E1 | LLM 服务不可用（本地模型未启动） | 未覆盖 | 定义降级策略：员工全部显示 offline？弹窗提示配置？ |
| E2 | 创建员工数量上限 | 成功指标写">= 5"，但上限未定义 | 需要明确：是否有上限？受限于什么（内存？LLM并发？） |
| E3 | 所有员工同时工作的资源消耗 | 架构评估提到了（R1/R4），但产品侧未做取舍 | 产品需定义：MVP阶段推荐多少员工？是否需要"员工调度策略"让用户选择优先级？ |
| E4 | 数据迁移和备份 | 完全未覆盖 | 本地应用的数据安全至关重要：导出/导入、备份恢复策略 |
| E5 | 多项目切换 | 长期目标提到但无细节 | 员工在项目A的记忆如何隔离/共享到项目B？ |

### 4. 跨文档一致性

- overview 成功指标"渲染帧率 >= 30fps"与 DESIGN-visual-style.md 的"Target frame rate: 30 FPS"一致
- overview 成功指标"3实例并行"与架构审核建议"Phase 1 先做单实例"存在矛盾——建议将MVP指标调整为"单实例可用"，V1.0 再要求三实例
- overview 的"桌面端应用"在 scope 中提及但未在 MVP 目标中体现——Electron 打包是 MVP 还是后续？需明确

overview 作为产品总纲，定位清晰、目标分层合理。主要不足是：缺少用户场景叙事、边界定义不充分、与子PRD和架构文档存在少量不一致。建议补充上述内容后作为正式基线。
