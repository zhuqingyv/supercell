# PRD-visual: 视觉与交互需求

## 1. 视觉风格定义

### 整体风格：Q版像素风 (Chibi Pixel Art)

- 角色比例：2-3 头身比例的 Q 版像素角色
- 分辨率：角色 32x32 或 48x48 像素基准，场景按需缩放
- 色调：明快柔和，办公室暖色调为主
- 参考：类似《星露谷物语》《Habitica》的像素风格，但更偏现代办公场景

### 调色板

| 用途 | 色值范围 |
|------|----------|
| 背景/地板 | 浅木色系 #E8D5B7 ~ #D4B896 |
| 墙壁 | 浅灰白 #F0EDE8 ~ #E5E0D8 |
| 家具 | 中性木色 + 科技灰 |
| 角色 | 鲜艳多彩，每个角色一个主题色 |
| UI 元素 | 与当前暗色主题保持统一 |

## 2. 虚拟办公室场景

等距视角 (Isometric) 或俯视视角 (Top-down) 的办公室场景：

```

│                虚拟办公室                      │
│                                               │
│   ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐       │
│   │ 工位 │  │ 工位 │  │ 工位 │  │ 工位 │       │
│   │  A   │  │  B   │  │  C   │  │  D   │       │
│   └──┬──┘  └──┬──┘  └──┬──┘  └──┬──┘       │
│      │        │        │        │            │
│   ┌──▼──┐  ┌──▼──┐  ┌──▼──┐  ┌──▼──┐       │
│   │角色A │  │角色B │  │角色C │  │角色D │       │

│                                               │

```

### 2.2 场景元素

| 元素 | 说明 | 交互 |
|------|------|------|
| 员工工位 | 包含桌子、电脑、椅子 | 点击进入工作空间 |
| 员工角色 | Q版像素小人 | 点击弹出信息卡片 |
| 电脑屏幕 | 工位上的显示器 | 点击查看员工聊天/工作 |
| 会议室 | 多人讨论区域 | 团队任务时角色聚集 |
| 茶水间 | 休息区域 | 空闲角色会移动到此 |
| 状态气泡 | 角色头顶的状态图标 | 悬停显示详情 |

## 3. 角色动画系统

### 3.1 骨骼动画方案

推荐使用 Spine 或 DragonBones 的免费替代方案：

**首选：Spine 2D** (如有授权) 或 **DragonBones** (开源免费)
- DragonBones: 开源骨骼动画工具，支持导出到 PixiJS/Phaser
- 像素风骨骼动画资源库参考：itch.io 上的 CC0 像素角色资源包

**降级方案：精灵帧动画 (Sprite Sheet)**
- 使用 Aseprite 制作帧动画
- 每个状态 4-8 帧
- 文件小，性能好，但表现力有限

### 3.2 角色动画状态

| 状态 | 动画描述 | 触发条件 |
|------|----------|----------|
| `idle` | 呼吸动效，偶尔眨眼 | 默认状态 |
| `working` | 面对电脑打字，屏幕有光效 | Worker 实例运行中 |
| `thinking` | 头顶出现"..."气泡，挠头 | MemoryOrganizer 运行中 |
| `chatting` | 头顶出现聊天气泡 | Communicator 运行中 |
| `walking` | 行走到目标位置 | 状态切换移动 |
| `celebrating` | 跳跃 + 星星特效 | 任务完成 |
| `error` | 头顶出现 "!" + 冒汗 | 任务出错 |
| `sleeping` | 趴在桌上 ZZZ | 离线/长时间空闲 |

### 3.3 动画优先级

当多个实例同时运行时，角色显示最高优先级的动画：
1. `error` (最高)
2. `working`
3. `chatting`
4. `thinking`
5. `idle` (最低)

头顶同时显示多个小图标表示多实例并行。

## 4. 员工状态可视化

### 4.1 状态指示器

每个员工角色身上有以下可视化元素：

```
     [状态图标]        ← 头顶气泡/图标
      (^_^)           ← 表情
     /角色\            ← 像素小人
    ─┤    ├─
     名字             ← 底部名称标签
    [进度条]           ← 当前任务进度（可选）
```

### 4.2 工位状态

| 工位元素 | 空闲时 | 工作时 |
|----------|--------|--------|
| 电脑屏幕 | 黑屏/屏保 | 亮起，显示代码/文档 |
| 桌面 | 整洁 | 出现文件/咖啡杯 |
| 椅子 | 空 | 角色坐在上面 |

### 4.3 多实例可视化

当员工三个实例同时运行时：

```
     [工作] [记忆] [聊天]    ← 三个小图标同时显示
         (^o^)
        /角色\
```

- 每个实例用不同颜色的小图标表示
- 工作：蓝色齿轮
- 记忆：紫色脑图标
- 聊天：绿色聊天气泡

## 5. 交互设计

### 5.1 员工卡片

点击员工角色，弹出信息卡片：

```
┌──────────────────────────┐
│  [头像]  张三             │
│          全栈开发工程师     │
│                           │
│  状态：工作中 🔵           │
│  当前任务：修复登录Bug     │
│  进度：75%                │
│                           │
│  [工作空间] [聊天] [管理]  │
└──────────────────────────┘
```

### 5.2 点击交互流程

```
虚拟办公室主界面
  │
  ├── 点击员工 → 弹出卡片
  │     ├── 点击 [工作空间] → 进入工作空间详情页
  │     ├── 点击 [聊天] → 打开与该员工的聊天窗口
  │     └── 点击 [管理] → 打开员工管理面板
  │
  ├── 点击员工电脑 → 查看员工"虚拟屏幕"
  │     ├── 聊天软件视图（看员工间通信）
  │     ├── 代码编辑器视图（看员工写代码）
  │     └── 终端视图（看命令执行）
  │
  ├── 点击会议室 → 查看当前团队讨论
  │
  └── 搜索栏 → 全局搜索员工工作空间
```

### 5.3 拖拽交互

- 拖拽员工到会议室 → 发起团队讨论
- 拖拽任务到员工 → 分配任务
- 拖拽文件到员工 → 让员工处理文件

### 5.4 右键菜单

右键点击员工：
- 分配任务
- 发消息
- 查看工作日志
- 调整优先级
- 解雇

## 7. 响应式设计

### 7.1 桌面端布局 (主要目标)

- 主区域：虚拟办公室场景（可缩放/拖动）
- 右侧/底部：活动日志 + 快捷操作
- 弹窗层：卡片、工作空间、聊天窗口

### 7.2 性能目标

| 指标 | 目标 |
|------|------|
| 场景渲染帧率 | >= 30fps (目标 60fps) |
| 角色动画 | 同屏 20 个角色流畅 |
| 首屏加载 | < 3s |
| 交互响应 | < 100ms |

### 2. 用户场景补充

**场景：办公室缩放和导航**
- 当员工数量超过屏幕可展示范围时，如何导航？
- 缩放的最小/最大比例？
- 是否支持键盘导航（如方向键平移、+/- 缩放）？
- 建议：定义缩放范围（0.5x - 3x）、平移边界、快捷键

**场景：员工工位自动分配**
- 新员工创建后如何分配工位？
- 用户能否手动调整工位布局？（拖拽员工到不同工位）
- 办公室空间不够了怎么办？（扩容？多楼层？）

**场景：无障碍访问**
- 像素风格的低对比度可能对视障用户不友好
- 状态完全依赖颜色（绿/蓝/黄/橙/灰红）——色盲用户如何区分？
- 建议：状态指示除颜色外增加形状/图案区分

### 3. 边界Case

| # | 边界场景 | 建议 |
|---|---------|------|
| V-E1 | 大量角色同时切换状态（如 Leader 分配批量任务） | 多个角色同时播放状态转换动画是否流畅？是否需要错开/简化？ |
| V-E2 | 等距视角下角色重叠 | 多个角色走向会议室时路径重叠，z-order 如何处理？ |
| V-E3 | 会议室容量 | 最多支持多少角色同时开会？场景布局图只画了一个会议室 |
| V-E4 | 通知气泡堆积 | 如果短时间内有大量通知（10+任务完成），如何避免气泡遮挡？ |
| V-E5 | 拖拽交互在触摸屏上 | 5.3 定义了拖拽交互，但在触摸屏（未来 Tauri 移动端）上如何适配？ |

### 4. 跨文档一致性

- PRD-visual 定义了 8 种动画状态（idle/working/thinking/chatting/walking/celebrating/error/sleeping），但 DESIGN-states.md 定义了 11 种状态，DESIGN-visual-style.md 定义了 11 种动画（idle/coding/designing/testing/writing/reviewing/meeting/chatting/break/memorizing/offline）。**PRD-visual 的状态粒度不够**，需与设计文档对齐到 11 种
- PRD-visual 的动画优先级（error > working > chatting > thinking > idle）与 DESIGN-states.md 的优先级（offline > meeting > working子状态 > chatting > memorizing > break > idle）**不一致**
- 性能目标"同屏 20 个角色"与 DESIGN-visual-style.md 的"Max simultaneous animated characters: 15"不一致——需统一
- 员工卡片设计（5.1）与 DESIGN-wireframes.md 第2节的卡片线框图一致，但 PRD 的卡片没有"最近活动"区域，而线框图有

### 5. 总体评价

视觉与交互需求覆盖完整，场景布局、动画状态、交互流程定义清晰。拖拽交互和右键菜单是很好的补充。主要问题：(1) 动画状态粒度与设计文档不一致（需对齐到 11 种）；(2) 性能指标数字需统一；(3) 缺少无障碍和缩放导航的考虑。
